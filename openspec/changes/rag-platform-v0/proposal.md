## Why

企业内部知识资产分散在文档、工单、代码与业务系统中，缺少统一的治理与检索入口，导致复用成本高、问答体验差、模型接入与成本不可控。需要一个以“企业空间”为租户边界的 RAG 管理平台，统一管理模型与知识库，并支撑后续持续演进。

## What Changes

- 新增企业空间（租户）管理：以企业空间作为唯一租户标识，隔离知识库与模型配置；工程首次启动时初始化管理员账号，并自动创建企业空间 `default`，后续所有操作默认基于该空间。
- 新增模型管理：以统一 Provider 抽象对接主流公有云/厂商模型服务（阿里云百炼、DeepSeek、智谱、Kimi 等），支持按企业空间配置与使用。
- 新增知识库管理：以 PostgreSQL 存储业务元数据；支持对接 Milvus（向量）与 Neo4j（图）作为检索与关系增强的底座；V0 以“可连接 + 最小读写闭环”为目标，具体细分能力在规格阶段补全。
- 新增基础工程骨架与开发编排：后端 FastAPI（Python 3.12+）+ SQLAlchemy + Pydantic + PostgreSQL；前端 Vue 3 + Vite（主流生态）；按约定的目录结构组织，并提供开发环境 docker-compose 编排。

## Capabilities

### New Capabilities

- `enterprise-space`: 企业空间（租户）生命周期与隔离策略；包含启动初始化（管理员账号 + `default` 企业空间）与幂等规则；所有资源均按企业空间隔离与审计。
- `model-management`: 模型 Provider 抽象、Provider/模型配置管理、凭据管理（仅写入不回显）与按企业空间隔离的调用上下文。
- `knowledge-base-management`: 知识库元数据管理与存储连接配置（PostgreSQL + Milvus + Neo4j）；V0 定义 Milvus/Neo4j 的最小可用闭环（可连接、可写入、可检索/可查询的最小路径）。

### Modified Capabilities

- （无）

## Tenant Isolation Strategy（企业空间隔离策略）

- 企业空间为唯一租户边界：所有与知识库、模型、连接配置相关的数据模型必须带有 `enterprise_space_id`（或等价外键/字段），并在数据库层建立必要的索引与约束。
- API 以企业空间为上下文：通过路径参数、请求头或上下文注入之一承载当前企业空间；任何查询/写入必须在服务层强制过滤企业空间，避免越权访问。

## Bootstrap Initialization（启动初始化）

- 工程首次启动必须完成初始化：创建管理员账号并创建默认企业空间 `default`；重复启动应具备幂等性（不会重复创建或破坏现有数据）。
- 初始化方式需可控：通过环境变量/启动参数指定管理员初始信息（如用户名/邮箱/初始密码或一次性口令），敏感信息不得在日志中明文输出。

## Impact

- 后端：新增租户/模型/知识库相关的数据库表、Pydantic schema、FastAPI 路由与依赖注入；增加 Milvus 与 Neo4j 的连接与最小操作封装；引入统一错误模型与鉴权/租户上下文中间件（或依赖项）。
- 前端：新增企业空间入口与基础管理页面骨架；模型管理与知识库管理的列表/创建/配置界面（V0 以管理能力为主）。
- 基础设施：开发环境新增 PostgreSQL、Milvus、Neo4j 的 docker-compose 编排；环境变量模板补充连接与凭据配置；Makefile 增加常用开发命令。

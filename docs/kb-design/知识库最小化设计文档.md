# 知识库最小化设计文档

## 1. 设计目标

基于当前工程现有架构，设计一套**最小可落地**的知识库方案，优先覆盖以下 3 大能力：

1. 知识库的创建与管理
2. 文档的上传、解析、分块与索引
3. 检索实现：先支持向量检索、关键词检索、混合检索

本次设计遵循以下原则：

- **最小化原则**：只做当前闭环必需能力，不提前引入复杂任务编排、图谱抽取、多模态、FAQ、RAPTOR 等增强能力。
- **复用现有架构**：复用当前 `enterprise_space` 多租户、`knowledge_base` 配置、`ai_model` 默认模型、`Milvus` 接入能力。
- **可演进原则**：表结构和接口为后续异步化、图数据库增强、重排序、对象存储预留空间。
- **先元数据、后能力增强**：先建立稳定的数据主线 `知识库 -> 文档 -> 分块 -> 检索`，高级能力后续叠加。

## 2. 与现有工程的衔接

### 2.1 直接复用的现有能力

1. **企业空间隔离**
   - 复用 `enterprise_space`、`membership`、`X-Enterprise-Space` 解析机制。
   - 所有知识库、文档、分块、检索请求均按 `enterprise_space_id` 隔离。

2. **现有知识库基础表**
   - 保留当前 `knowledge_base`、`neo4j_connection`。
   - 将当前 `milvus_connection` 更名为 `knowledge_milvus_config`，用于描述“某个知识库绑定的 Milvus 配置与 collection 信息”，而不是单纯的连接对象。
   - `Neo4jConnection` 和 `graph_db_enabled` **已废弃为主路径**；新图知识库请使用 `kb_type=lightrag` + 平台 Neo4j + LightRAG（见 OpenSpec `lightrag-graph-kb`）。

3. **现有模型管理能力**
   - 复用 `ai_model`、`ai_model_default`。
   - 知识库向量化优先使用企业空间默认 `embedding` 模型。
   - 后续如需要知识库级别覆盖，可支持单独指定 `embedding_model_id`。

4. **现有技术栈**
   - 元数据：PostgreSQL
   - 向量检索：Milvus
   - API：FastAPI
   - ORM：SQLAlchemy
   - 前端：Vue 3 + Pinia + Router

### 2.2 本次明确不做

以下能力本期只预留，不实现：

- 图数据库写入与图检索
- 文档异步任务队列/Celery/RQ
- 多文件批量上传
- OCR / 图片解析 / 表格增强解析
- Rerank 重排序
- 跨知识库检索
- 问答生成（本期只做检索，不直接生成回答）
- 文档版本管理、回滚、导出

## 3. 业务范围与最小闭环

### 3.1 知识库管理

最小支持：

- 创建知识库
- 编辑知识库基础信息
- 删除知识库（软删除）
- 配置向量存储（当前固定 Milvus）
- 测试向量存储连通性（当前固定 Milvus）
- 查看知识库统计（文档数、分块数、索引状态）

### 3.2 文档管理

最小支持：

- 上传单个文档
- 支持 `txt`、`md`、`pdf`
- 文档解析为纯文本
- 按固定大小进行分块（支持 overlap）
- 生成关键词检索索引
- 生成向量并写入 Milvus
- 查看文档列表、详情、处理状态
- 删除文档并同步删除其 chunk 与向量索引
- 手动重建索引

### 3.3 检索能力

最小支持 3 种模式：

- `vector`：向量检索
- `keyword`：关键词检索（PostgreSQL Full Text Search）
- `hybrid`：向量 + 关键词混合召回与融合排序

- 每个知识库使用一个独立 collection。

## 5. 文件存储设计（最小版）

### 5.1 存储方式

最小版采用**本地文件存储**，目录建议：

`backend/storage/knowledge_files/{space_id}/{kb_id}/{document_id}/original.{ext}`

原因：

- 当前工程尚无对象存储基础设施。
- 最小版先解决上传、解析、索引闭环。
- 后续可无缝替换为 MinIO / S3，只需保留 `storage_type + storage_path` 抽象。

### 5.2 支持文件类型

首期最小支持：

- `.txt`
- `.md`
- `.pdf`

本期不支持：

- `.docx`
- 图片
- 音视频
- OCR

## 6. 文档处理流程设计

## 6.1 上传与入库流程

`上传文档 -> 保存文件 -> 创建 document 记录 -> 解析文本 -> 分块 -> 写 chunk 表 -> 生成向量 -> 写 Milvus -> 更新 document 状态`

### 6.1.1 处理步骤

1. 校验知识库是否存在、是否属于当前企业空间。
2. 校验文件类型、文件大小。
3. 保存原始文件到本地目录。
4. 写入 `knowledge_document`，状态为 `uploaded`。
5. 解析文本：
   - `txt`：直接读取
   - `md`：直接读取并去除无意义控制字符
   - `pdf`：使用页级文本抽取
6. 更新状态为 `parsing`。
7. 使用固定规则分块，更新状态为 `chunking`。
8. 批量写入 `knowledge_chunk`。
9. 如果知识库启用了向量检索，则：
   - 获取 embedding 模型
   - 生成各 chunk 向量
   - 写入 Milvus collection
   - 将 chunk 的 `vector_status` 标记为 `indexed`
10. 文档状态更新为 `indexed`；失败则标记 `failed` 并写入 `error_message`。

### 6.1.2 分块策略

最小版采用**固定大小字符分块 + overlap**：

- 默认 `chunk_size = 500`
- 默认 `chunk_overlap = 80`
- 按自然段优先切分，超长再按字符补切
- 保留 `page_no`、`heading_path`、`chunk_index`

说明：

- 初版不引入 token 级切分、父子分块、语义分块。
- 后续如接入 tokenizer，可将字符切分升级为 token 切分。

## 6.2 删除与重建索引

### 删除文档

删除文档时执行：

1. 标记 `knowledge_document.status = deleted`
2. 删除该文档下所有 `knowledge_chunk`
3. 删除 Milvus 中对应 `chunk_uid` 的向量
4. 删除本地原始文件

### 重建索引

最小版提供手动重建：

- 清空文档当前 chunk 与向量索引
- 重新执行解析、分块、向量化

## 7. 检索设计

## 7.1 检索输入

最小检索请求字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| query | string | 查询语句 |
| mode | string | `vector/keyword/hybrid` |
| top_k | int | 返回结果条数，默认 5 |
| score_threshold | float nullable | 最低得分阈值 |
| document_ids | array<int> nullable | 限定文档范围 |
| include_disabled | bool | 初版固定 false，可不暴露 |

## 7.2 向量检索

### 依赖

- 知识库必须开启 `vector_db_enabled`
- 知识库必须存在可用 `knowledge_milvus_config`
- 当前企业空间必须存在默认 `embedding` 模型，或知识库设置了 `embedding_model_id`

### 流程

1. 读取 embedding 模型配置
2. 生成 query embedding
3. 在知识库对应 Milvus collection 中搜索 topN
4. 取回对应 `chunk_uid`
5. 查询 PostgreSQL 中对应 chunk 详情
6. 返回标准结果结构

### 返回字段

| 字段 | 说明 |
|---|---|
| chunk_id | chunk 主键 |
| chunk_uid | 稳定标识 |
| document_id | 文档 ID |
| document_name | 文档名 |
| chunk_index | 分块序号 |
| content | 分块正文 |
| score | 最终得分 |
| vector_score | 向量原始得分 |
| keyword_score | 关键词得分，向量模式下为空 |
| metadata | 页码、标题路径等 |
| citation | 用于前端引用展示 |

## 7.3 关键词检索

### 技术选型

最小版采用 **PostgreSQL Full Text Search**，不引入 ES / OpenSearch。

原因：

- 当前工程已稳定使用 PostgreSQL
- 最小实现成本低
- 对中小规模知识库可接受

### 流程

1. 将 query 转为 `tsquery`
2. 在 `knowledge_chunk.search_vector` 上执行检索
3. 使用 `ts_rank_cd` 排序
4. 结合过滤条件（知识库、文档、企业空间）返回结果

说明：

- 初版关键词检索主要面向中文短文本与英文关键字匹配场景。
- 如后续需要更强分词与 BM25，可扩展为 pg_bm25 或独立搜索引擎。

## 7.4 混合检索

### 设计目标

混合检索不是重新造一套索引，而是：

- 向量检索负责语义召回
- 关键词检索负责字面命中
- 应用层对两路结果做去重、归一化、融合排序

### 流程

1. 向量检索召回 `top_k * 4`
2. 关键词检索召回 `top_k * 4`
3. 用 `chunk_uid` 合并结果
4. 分别归一化 `vector_score` 与 `keyword_score`
5. 最终得分：

`final_score = 0.5 * vector_score_norm + 0.5 * keyword_score_norm`

6. 按最终得分排序并截取 `top_k`

说明：

- `0.5 / 0.5` 为默认权重（中文企业语料偏均衡，兼顾语义与精确关键词命中）；可放入知识库配置按场景调整。
- 本期不做 rerank；后续可在混合召回后追加 rerank 模型。

## 11. 关键设计决策

### 决策 1：关键词检索使用 PostgreSQL FTS

原因：

- 当前工程已依赖 PostgreSQL
- 实现成本最低
- 能满足最小版关键词检索需求

### 决策 2：文档处理首期同步执行

原因：

- 省去任务系统与队列依赖
- 更适合当前最小闭环
- 通过 `status` 字段为后续异步化预留空间

### 决策 3：文件先落本地磁盘

原因：

- 当前无对象存储设施
- 与最小化原则一致
- 后续可以平滑升级为 MinIO/S3

### 决策 4：图数据库只保留连接能力

原因：

- 当前需求主线是知识库创建、文档处理、向量/关键词/混合检索
- 图谱抽取与图检索不属于本期最小闭环

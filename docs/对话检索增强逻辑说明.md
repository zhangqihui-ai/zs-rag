# 对话页「检索增强」逻辑说明

本文说明：**工作台 `/chat` 发送消息时**，系统如何把知识库检索结果注入大模型上下文、如何生成引文，并与 **知识检索测试页 `/retrieval`** 的行为对比。便于理解「同一知识库、同一问题，两边结果条数或文档不一致」等现象。

---

## 1. 总览：一轮对话里发生了什么

以 **Web 对话（SSE 流式）** 主路径为例，核心代码在 `backend/app/services/chat_service.py` 的 `iter_chat_stream_events`。

单轮用户发送消息后，大致顺序为：

1. **持久化用户消息**到当前会话。
2. **用本轮用户输入**作为唯一检索查询串，调用 `_retrieve_knowledge_block_and_citations`，从当前对话配置的 `knowledge_base_ids` 中拉取片段。
3. 将返回的文本块拼成 **知识块 `kb_block`**，并按配置追加引文角标提示 `_CITATION_HINT_SUFFIX`（若开启「显示引文」）。
4. 读取本会话最近 **20 条**消息作为 `messages`，通过 `inject_system_into_messages` 把 **`{knowledge}`** 注入系统提示词（模板里的占位符被替换为 `kb_block`）。
5. 将完整 `messages` 发给 **对话大模型**流式生成，并把本步检索得到的 **citations** 与助手最终回复一并落库。

> **要点**：检索所用的 **query 是本轮用户输入原文**（截断至 2000 字符），**不会**把历史多轮对话拼接进检索 query。  
> 历史消息**只影响**大模型综合回答（系统提示里写了「回答需要考虑聊天历史」），**不直接改变**向量/Milvus 或关键词检索用的字符串。

（OpenAI 兼容 `chat/completions` 另一条路径中，`build_completion_messages` 同样对 `_retrieve_knowledge_block_and_citations` 传入 **最后一条用户话**或 **question** 字段，而不是整段历史摘要。）

> 多轮 **会话历史** 如何加载、`session_id` 与 `messages` 的关系，见 [`会话多轮历史实现说明.md`](./会话多轮历史实现说明.md)。

## 2. `_retrieve_knowledge_block_and_citations`：对话专用检索封装

位置：`backend/app/services/chat_service.py`。

### 2.1 入参

| 参数 | 含义 |
|------|------|
| `enterprise_space_id` | 企业空间 |
| `knowledge_base_ids` | 对话配置中的知识库 ID 列表（与设置里「知识库上下文」多选一致） |
| `query` | 检索用查询串（对话里 = 本轮用户输入） |
| `max_chunks` | 默认 **12**：写入上下文 + 引文列表时**最多采用**的条数（见下） |

### 2.2 每个知识库请求的条数 `per_kb`

```text
per_kb = max(3, min(8, max_chunks // len(knowledge_base_ids) + 2))
```

- **只挂载 1 个知识库**、`max_chunks=12`（默认）时：  
  ├ `12 // 1 + 2 = 14`  
  ├ `min(8, 14) = 8`  
  └ `per_k = max(3, 8) = **8**`  
  即：对每个库调用底层检索时，请求 **top_k = 8**。

- 挂载多个库时，`per_kb` 会变小，且多库结果会 **按分数合并排序** 后再取全局前 `max_chunks` 条（默认最多 12 条）。

### 2.3 调用底层检索时传入的 `KnowledgeSearchRequest`

对话里**仅显式**传入：

- `query = query[:2000]`
- `top_k = per_kb`（单库时一般为 **8**）

**未**传入（均为缺省 / 由知识库配置决定）：

| 字段 | 对话中的行为 |
|------|----------------|
| `mode` | `None` → 使用 **`KnowledgeBase.default_retrieval_mode`**（vector / keyword / hybrid） |
| `score_threshold` | `None` → 若库上配置了 **`default_score_threshold`** 则用之（**平台默认 0.5**），否则不过滤 |
| `vector_weight` | `None` → 混合模式下读 **`config.retrieval.vector_weight`**（**平台默认 0.3**），全文权重为 `1 - vector_weight` |
| `document_ids` | `None` → **不**按文档 ID 限域，库内**全部已索引文档**参与检索 |

底层实现见 `backend/app/services/retrieval_service.py` 的 `search_knowledge_base`。

### 2.4 引文与上下文条数

- 对每个库拿到 `results` 后，按 **`chunk_uid` 去重**，压平为 `(score, item, kb_id)`，按 **score 降序**排序。
- 取前 **`max_chunks`（默认 12）** 条，格式化为发给模型的「知识库」正文，并生成 **`ref` 1…N** 的 `citations` 列表（与前端角标 `[1]`、`[2]` 对应）。

多库时每条引文携带正确的 **`knowledge_base_id`**（便于前端按库拉切片详情）。

## 3. `search_knowledge_base`：与「检索测试」共用

路由：`POST /api/v1/knowledge-bases/{kb_id}/search`  
请求体：`KnowledgeSearchRequest`（`backend/app/schemas/retrieval.py`）。

### 3.1 三种模式（由 `mode` 或库默认值决定）

- **vector**：Milvus 向量检索，得分来自向量距离。
- **keyword**：PostgreSQL `tsvector` + `plainto_tsquery('simple', query)`，必要时 `keyword_text ILIKE` 兜底。
- **hybrid**：同时取向量候选与关键词候选（内部先各召回 `top_k × 4` 条），**分路过滤**后按权重融合，再排序截断到 `top_k`。详见下文 **§3.3、§5**（含 **§5.2 各自归一化**）。

### 3.3 混合检索流程（权重模式）

```text
用户问题
  ├─ 向量路：Milvus Top (K×4) → 分路 Score 阈值过滤 → 候选池 A
  └─ 全文路：OpenSearch / PG Top (K×4) → 分路 Score 阈值过滤 → 候选池 B
        ↓
  以全文候选为主干，叠加向量路头部命中（仅高置信向量参与加权）
        ↓
  融合分 = vector_weight × 向量归一化分 + (1 - vector_weight) × 全文归一化分
        ↓
  按融合分排序 → 取 Top K 返回
```

- 向量路与全文路的**原始分数刻度不同**（向量约 0～1 的余弦相似度；全文为加权关键词分），**不能**直接拿同一个绝对值比较，因此阈值在**各路内部**先做 min-max 归一化再判断。
- 若全文路最高分低于 **1.5**（仅命中「需要」「对方」等泛词），视为**该库无有效全文命中**，不参与融合；此时仅当向量 Top1 ≥ **0.45** 才单向量兜底，否则返回空（避免跨领域误召回，如医保库问刑法问题）。

## 4. 知识库默认检索配置（平台级）

新建知识库与已有库（迁移 `023_kb_default_retrieval`）统一采用以下默认值，持久化在：

- 表字段：`knowledge_base.default_score_threshold`
- JSON：`knowledge_base.config.retrieval`

| 配置项 | 平台默认值 | 存储位置 | 说明 |
|--------|------------|----------|------|
| 检索模式 | **hybrid**（混合检索） | `default_retrieval_mode` | 同时走向量 + 全文 |
| Top K | **5** | `default_top_k` | 最终返回条数上限 |
| 向量相似度权重 | **0.30** | `config.retrieval.vector_weight` | 语义分支权重 |
| 全文（关键词）权重 | **0.70** | `1 - vector_weight` | 由界面自动计算 |
| Score 阈值 | **开启，0.5** | `default_score_threshold` + `config.retrieval.score_threshold_enabled` | 混合检索分路过滤 |

代码常量见 `backend/app/core/knowledge_retrieval_defaults.py`；创建库时由 `apply_retrieval_defaults_to_payload` 自动补齐。

**为何默认 0.3 / 0.7：** 中文法条、医保制度、公文类语料中，专名与条款词面命中比纯语义向量更可靠；向量权重过高易召回「语义沾边但主题无关」的片段。

**单库可覆盖：** 在知识库详情 → 检索设置中修改并保存，只影响该库；对话与检索测试页读取库级配置，无需改代码。

## 5. 混合检索策略：Score 阈值详细说明

对应界面：**知识库详情 / 检索测试页 → 混合检索 → 权重设置 → Score 阈值**（开关 + 0～1 滑条，旁有 `?` 说明）。

### 5.1 这个阈值过滤的是什么？

**不是**直接拿向量原始分（如 0.038）或全文原始分（如 4.5）与 0.5 比较。

在**混合检索 + 阈值开启**时，流程为：

1. **向量路**：对当次召回的向量候选，在**本路内部**做 min-max 归一化（映射到 0～1），仅保留归一化分 **≥ 阈值** 的片段进入融合池。
2. **全文路**：同理，对全文候选在本路内归一化，仅保留 **≥ 阈值** 的片段。
3. **额外硬底线（与开关无关，混合模式始终生效）**：
   - 全文最高分 **< 1.5**：视为无有效命中（多为只命中「需要」「对方」等泛词），全文路整体不进融合。
   - 向量仅对**本路 Top 25% 且 ≥ 最高分×55%** 的命中参与加权，避免弱向量把全文噪声顶到前面。
4. **融合**：对通过过滤的片段按 **向量权重 / 全文权重** 计算融合分并排序。
5. **输出**：取 Top K；若融合后仍不足，返回条数可能少于 K（包括 **0 条**，表示库内无足够相关片段）。

### 5.2 「各自归一化」是什么意思？如何计算？

**各自归一化**指：向量路和全文路**分开**用各自当次召回的候选算分、做 **Min-Max 缩放**到 0～1，再**分别**与 Score 阈值比较；两路原始刻度不同，**不能混在一条标尺上比**。

#### 为什么两路不能共用同一个绝对阈值？

| 路 | 原始分大致范围 | 示例 |
|----|----------------|------|
| **向量路** | Milvus 余弦相似度，常见 0.03～0.70 | 0.038、0.384 |
| **全文路** | 加权关键词分，常见 0.35～4.5+ | 0.35（只命中「需要」）、4.5（命中「防卫+杀人」） |

若用**同一个 0.5** 去比**原始分**：

- 向量 0.038 < 0.5 → 几乎全被滤掉，阈值对向量路失效；
- 全文 4.5 > 0.5 → 几乎全保留，阈值对全文路太松。

因此必须在**向量候选内部**、**全文候选内部**各做一次 0～1 映射，再用 0.5 过滤。

#### 计算公式（Min-Max 归一化）

对**某一路**当次召回的所有候选，设原始分为 \(s_i\)，该路最高分为 \(s_{\max}\)，最低分为 \(s_{\min}\)：

\[
\text{归一化分}_i = \frac{s_i - s_{\min}}{s_{\max} - s_{\min}}
\]

- 本路**最高**的那条 → **1.0**
- 本路**最低**的那条 → **0.0**
- 其余落在 0～1 之间，按原始分线性插值

若该路所有候选**同分**（\(s_{\max} = s_{\min}\)）：

- 一般视为本路「并列第一」，归一化分均为 **1.0**；
- **例外（全文路）**：若同分且最高分 **< 1.5**（如全是 0.35，只命中泛词「需要」），则整路视为无效，归一化分置 **0**，不参与融合（见 §5.1 硬底线）。

实现见 `backend/app/services/retrieval_service.py` 中 `_normalize_scores`、`_filter_hybrid_channel_candidates`。

#### 举例 1：全文路（刑法库，阈值 0.5）

当次召回 4 条，**原始全文分**如下：

| 片段 | 原始全文分 |
|------|-----------|
| 正当防卫法条 | **4.5** |
| 罚金条款 | 3.35 |
| 缓刑条款 | 3.0 |
| 附加刑 | 1.85 |

本路 \(s_{\max}=4.5\)，\(s_{\min}=1.85\)，分母 \(=2.65\)：

| 片段 | 归一化计算 | 归一化分 | ≥ 0.5？ |
|------|-----------|---------|--------|
| 正当防卫 | (4.5−1.85)/2.65 | **1.0** | ✅ 进入融合 |
| 罚金 | (3.35−1.85)/2.65 | **0.57** | ✅ |
| 缓刑 | (3.0−1.85)/2.65 | **0.43** | ❌ 被阈值挡掉 |
| 附加刑 | (1.85−1.85)/2.65 | **0.0** | ❌ |

#### 举例 2：向量路（阈值 0.5）

当次召回 5 条，**原始向量分**如下：

| 片段 | 原始向量分 |
|------|-----------|
| A | **0.068** |
| B | 0.061 |
| C | 0.060 |
| D | 0.055 |
| E | 0.038 |

本路 \(s_{\max}=0.068\)，\(s_{\min}=0.038\)，分母 \(=0.030\)：

| 片段 | 归一化分 | ≥ 0.5？ |
|------|---------|--------|
| A | 1.0 | ✅ |
| B | 0.77 | ✅ |
| C | 0.73 | ✅ |
| D | 0.57 | ✅ |
| E | **0.0** | ❌ |

要点：**原始 0.038 归一化后是 0.0，不是 1.0**——这就是「不是拿 0.038 直接和 0.5 比」的含义。

#### 举例 3：全文路全部同分（医保库误召回）

医保库问刑法，多条片段都只命中泛词「需要」，**原始全文分全是 0.35**：

- 最高分 0.35 **< 1.5** → 整路判定为**无有效全文命中**，直接返回空，**不做**「同分全变 1.0」的归一化；
- 从而避免出现「弱命中因同分被抬成满分、全部通过阈值」的问题。

#### 归一化在流程中出现两次（用途不同）

```text
① 分路过滤（Score 阈值）
   向量路：本路 Min-Max → 与阈值比较 → 留下候选子集
   全文路：本路 Min-Max → 与阈值比较 → 留下候选子集

② 权重融合（对通过 ① 的片段）
   再次对向量分、全文分分别 Min-Max
   融合分 = vector_weight × 向量归一化分 + (1 − vector_weight) × 全文归一化分
```

界面结果卡片底部的 **「向量 / 全文」** 展示的是**原始分**；列表排序用的 **score** 是**融合分**。Score 阈值只作用于步骤 ① 的分路归一化分，与卡片 footer 上的原始数字**不是同一刻度**。

### 5.3 阈值刻度与界面显示

| 场景 | 含义 |
|------|------|
| 阈值 **0.5**（平台默认） | 各路内至少达到「本路中等偏上」才进融合；偏**精准**，跨库误召回少 |
| 阈值 **0.3**（可手动调低，见界面滑条） | 保留更多本路弱命中，**召回更宽**；相关边界问题可能多 1～2 条，噪声也会增加 |
| 阈值 **关闭** | 不做用户阈值过滤；混合检索仍启用**内置相对阈值**（全文 ≥ 最高分×45% 且 ≥ 1.5 等） |

### 5.4 典型现象

| 现象 | 原因 |
|------|------|
| 刑法库问防卫，Top1 为「正当防卫」法条 | 全文「防卫」权重高，通过阈值后融合分领先 |
| 医保库问刑法，**0 条** | 全文仅命中泛词「需要」（0.35 < 1.5），向量 Top1 < 0.45，双路均无效 |
| 开启 0.5 后只有 1～2 条 | 正常；弱相关被分路阈值挡在融合池外 |
| 纯**向量检索**模式下阈值几乎无效 | 向量原始分常远小于 0.5，**阈值主要服务于混合检索** |

### 5.5 参数搭配建议

| 参数 | 推荐 | 说明 |
|------|------|------|
| 向量权重 | **0.30** | 制度 / 法条 / 医保政策库默认 |
| 全文权重 | **0.70** | 与向量权重之和为 1 |
| Score 阈值 | **默认开启 0.5** | 生产环境平衡精度与召回 |
| 召回偏少时 | 降至 **0.35～0.4** | 或暂时关闭阈值做对比测试 |
| Top K | **5** | 对话侧单库实际可能拉到 8（见 §2.2） |

## 6. 与对话的差异（为何同一库、同一问法，结果「看起来不一样」）

| 项目 | 检索测试页 `/retrieval`（默认行为） | 对话 `/chat` |
|------|--------------------------------------|--------------|
| **top_k** | 界面可调，重置默认常为 **5** | 单库时 **`per_k` 多为 8**（由上式算出），再被 `max_chunks=12` 截断 |
| **score_threshold** | 可手动填 | 一般**不**在对话里传参，只跟 **知识库 `default_score_threshold`** |
| **mode / vector_weight** | 页面可选 | 跟 **知识库默认**（除非以后对话侧扩展配置） |
| **document_ids** | 可填，仅检索指定文档 | **始终不传** → 全库文档参与 |

因此常见情况之一：

- 测试页 **top_k=5**：只看到分数最高的 5 条。
- 对话 **每库拉 8 条**再注入：第 **6～8** 名可能进入模型上下文，测试页不会出现。

这不是「对话偷偷挂了第二个知识库」的必然结果；**仅挂一个库时**，排序靠后的片段仍可能来自**同一库内的其它文档**。

## 7. 系统提示词与 `{knowledge}`

默认系统模板（可被对话配置覆盖）中包含：

- 「总结知识库…」「当无关时要提示未找到」
- 「**回答需要考虑聊天历史**」
- `以下是知识库：{knowledge} 以上是知识库。`

`inject_system_into_messages` 会：

- 若消息列表首条已是 `system`：把其中 **`{knowledge}`** 替换为本轮检索拼出的 `kb_block`；
- 否则插入/合并系统消息。

因此：**检索内容只由本轮 query 决定**；**回答是否结合历史**由模型在已注入上下文的条件下自行组织，但 **引文列表的数据源仍是本轮检索返回的片段**，不是历史里曾经出现过的文档自动再检索一遍。

---

## 8. 可调优方向（产品 / 运维）

1. **对齐条数**：若希望对话与测试一致，可调整对话侧 `per_kb` / `max_chunks` 与测试 **top_k** 一致。  
2. **知识库级阈值**：默认 **0.5** 已写入所有库；召回不足时可单库降至 **0.3～0.4**（与界面滑条一致）。  
3. **拆库或限域**：敏感场景单独建库；未来若开放对话级 `document_ids` 可进一步限域。  
4. **混合权重**：中文制度类库维持 **0.3 / 0.7**；英文技术文档库可试提高到 **0.4～0.5** 做 A/B。

## 10. 小结

- **对话检索增强** = 每轮用**当前用户句**检索各库 → 合并排序 → 最多 **12** 条写入 `{knowledge}`，并生成 **citations**。  
- **平台默认**：混合检索、Top K=5、**向量 0.3 / 全文 0.7**、**Score 阈值开启 0.5**；新建库自动继承。  
- **Score 阈值（混合）** = 向量路与全文路**分别 Min-Max 归一化**（§5.2）后过滤，再融合；不是用原始向量分直接比 0.5。  
- **与检索测试不一致**的常见原因：**top_k（对话单库常 8 vs 测试 5）**、阈值与权重是否一致、全库多文档下排序靠后的片段。

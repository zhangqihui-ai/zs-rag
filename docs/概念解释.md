## **1.概念解释**


| 参数                | 含义                                                                      |
| ----------------- | ----------------------------------------------------------------------- |
| **chunk_size**    | 分块的最大大小（即一个chunk最多包含多少单位）。RAGFlow用 **token数**，WeKnora用 **字符数**，Yuxi兼容两者 |
| **chunk_overlap** | 相邻chunk之间的重叠部分大小，确保跨块边界的语义不丢失。重叠区域让检索到某个chunk时也能获取到前文/后文的上下文            |


### **1.1 RAGFlow**


| 参数            | 参数名                  | 默认值        | 单位     | 范围              |
| ------------- | -------------------- | ---------- | ------ | --------------- |
| chunk_size    | `chunk_token_num`    | **512**    | token数 | 1 ~ 2048（API限制） |
| chunk_overlap | `overlapped_percent` | **0**（无重叠） | 百分比(%) | 0 ~ 99          |


特殊场景默认值：

- `naive.py` 通用：`chunk_token_num=512`, `overlapped_percent=0`
- `paper.py` 论文：`chunk_token_num=512`
- `manual.py` 手册：`chunk_token_num=512`
- `book.py` 书籍：`chunk_token_num=512`（通过 `naive_merge` 传参默认128）
- 文件解析服务 `file_service.py`：`chunk_token_num=16096`（大文件）
- 文档解析服务 `document_service.py`：`chunk_token_num=4096`
- API创建KB默认：`chunk_token_num=512`
- Firecrawl工具：`chunk_size=1000`, `chunk_overlap=200`

**重叠实现**：`overlapped_percent` 是百分比，如设为20，则重叠 = `chunk_token_num * 20% = 102 tokens`。代码逻辑：当chunk达到 `chunk_token_num * (100 - overlapped_percent) / 100` 时就开始新chunk，并将上一个chunk末尾的部分拼接到新chunk头部。

## **2.父子分块（Parent-Child Chunking）**

### **核心思想**

传统分块有一个矛盾：**小块检索精度高，但上下文不够；大块上下文完整，但检索噪音多**。

父子分块通过**将同一文本切出两层粒度**来解决这个矛盾：



文档首先被分割成较大的父级块，每个父级块都保持相对完整的语义单元，以确保逻辑和背景的完整性。然后，每个父级块可以进一步细分为多个子级块，以实现精确检索。在检索过程中，系统首先基于子级块定位最相关的文本片段，同时自动关联并检索其父级块。这种方法在保持高检索相关性的同时，也为生成阶段提供了充足的语义背景。



例如，在处理*合规手册*时，用户查询“违约责任”时，系统可能会精确检索到一条子条款：“违约罚款为合同总价值的20%”，但由于缺乏上下文，无法明确该条款适用于“轻微违约”还是“重大违约”。利用父子条款分割机制，系统会返回该子条款及其父条款，父条款包含完整的条款内容。这使得LLM能够基于更广泛的上下文做出准确判断，避免误解。



通过“精确定位+上下文补充”的双层结构，确保了检索的准确性，同时显著提高了生成答案的可靠性和完整性。



```
┌─────────────────────────────────────────┐
│  父分块 (Parent Chunk) — 4096字符        │  ← 不参与向量索引，仅提供上下文
│  ┌───────────┐ ┌───────────┐ ┌────────┐ │
│  │ 子分块1    │ │ 子分块2    │ │ 子分块3 │ │  ← 参与向量索引和检索
│  │ 384字符   │ │ 384字符   │ │ 384字符│ │
│  └───────────┘ └───────────┘ └────────┘ │
└─────────────────────────────────────────┘

```

### **工作流程**

1. **创建**：大文本先切出父分块（大），再将父分块切成子分块（小）
2. **索引**：只对**子分块**做向量化，存入向量库
3. **检索**：用户查询 → 匹配到子分块 → 通过 `ParentChunkID` 找到对应父分块
4. **返回**：将**父分块**作为上下文送给LLM，而不是子分块

### **对比传统方式**


|            | 传统分块       | 父子分块          |
| ---------- | ---------- | ------------- |
| **检索粒度**   | 块大小固定，粒度单一 | 子块小 → 检索精准    |
| **上下文完整性** | 块太小则上下文断裂  | 父块大 → 上下文完整   |
| **向量索引量**  | 所有块都索引     | 只索引子块，父块不索引   |
| **存储开销**   | 1份         | 2份（但父块不占向量索引） |


### **举个具体例子**

假设一段法律条文：

> **第三条** 本法所称用人单位，是指企业、个体经济组织、民办非企业单位等组织......（后续2000字详细解释）

- **传统**：切成4个512字符块，第2块检索命中时，LLM看不到"第三条"这个标题上下文
- **父子**：子块1命中 → 返回父块（含完整第三条2000字），LLM获得完整法条上下文

### **WeKnora 的实现**

WeKnora 是三个项目中**唯一原生支持父子分块**的：



```
ChunkTypeText       ("text")        → 子分块，参与向量索引
ChunkTypeParentText ("parent_text") → 父分块，不参与向量索引，仅提供上下文

```

关键配置：

- `ParentChunkSize` = **4096** 字符
- `ChildChunkSize` = **384** 字符
- 子分块通过 `ParentChunkID` 字段关联到父分块

### **RAGFlow 的类似思路**

RAGFlow 虽然没有显式的"父子分块"类型，但通过 `content_ltks`（低权重关键词）和 `naive_merge` 的上下文合并机制，以及手动/法律/书籍等解析器的**层级分块**，在效果上也能保留上下文完整性。只是不如 WeKnora 的显式父子设计灵活。

分段：
[https://docs.dify.ai/zh/use-dify/knowledge/create-knowledge/chunking-and-cleaning-text](https://docs.dify.ai/zh/use-dify/knowledge/create-knowledge/chunking-and-cleaning-text)  

导入知识库的文档会被拆分为较小的片段，称为**分段**。分段的概念类似于将一本大书整理成章节和段落——你无法在一大块文本中快速找到特定信息，但组织良好的章节能使检索变得高效。当用户提出问题时，系统会在这些分段中搜索相关信息，并将其作为上下文提供给 LLM。如果没有分段，每次查询都需要处理整个文档，这将既缓慢又低效。**关键分段参数**

- **分隔符**：文本被拆分的字符或字符序列。例如，`\n\n` 在段落换行处拆分，`\n` 在行换行处拆分。
分隔符在分段过程中会被移除。例如，使用 `A` 作为分隔符会将 `CBACD` 拆分为 `CB` 和 `CD`。为避免信息丢失，请使用文档中不会自然出现的非内容字符。
- **分段最大长度**：每个分段的最大字符数。超过此限制的文本将被强制拆分，无论分隔符设置如何。

## **选择索引方式**

正如搜索引擎通过高效的索引算法匹配与用户问题最相关的网页内容，索引方式是否合理将直接影响 LLM 对知识库内容的检索效率以及回答的准确性。提供**高质量**与**经济**两种索引方式，其中分别提供不同的检索设置选项。
